NVIDIA's 15× speculative-decoding trick, on a $570 GPU — and an 8-year-old one

NVIDIA's dFlash post shows up to 15× higher throughput on eight DGX B300s. Almost nobody has those. One result genuinely surprised us.

Every speculative run's output is byte-identical to its baseline — speculative decoding changes speed, not answers.

The claim, and the honest question

In NVIDIA's dFlash announcement, a block-diffusion drafter pushes gpt-oss-120b to roughly 15× higher throughput across eight Blackwell-Ultra GPUs. It's a clever result — but it lives in a world of 8× DGX B300 nodes, NVFP4, and TensorRT-LLM.

The core trick, though, is not datacenter-only. Speculative decoding ships today in llama.cpp, runs on Windows, and works on any GPU. So how much of that speedup survives at home? We tested two cards eight years and one price-bracket apart.

How speculative decoding works (the 60-second version)

A language model normally emits one token per forward pass. Each pass drags the whole multi-gigabyte model through memory to produce a single token — that's why local generation is slow, and why it's memory-bound, not compute-bound. Speculative decoding attacks exactly that:

The win is leverage: you still pay for the big model, but one expensive pass now ratifies several tokens instead of one — if the draft guesses well. That "if" is the whole story: speedups rise and fall with the acceptance rate, which is why predictable text (code, math) wins and free-form prose doesn't.

The surprise: speculation has overhead, and it can backfire

This is the part worth bookmarking. Speculative decoding isn't free — every draft-then-verify cycle carries overhead, and if the target model is already fast, that overhead can cost more than speculation saves. We hit it hard: on Blackwell's quick 7B (~85 tok/s), the stock CUDA build's draft speculation collapsed to 0.27× of baseline — a 3–4× slowdown — at perfectly healthy acceptance rates. Yet the same CUDA build, same card, pointed at a slower 14B (~42 tok/s) delivered up to 1.9×, because the heavier model gives the overhead something to amortize against. The eight-year-old Pascal card never collapsed on the 7B either — its 7B only runs ~50 tok/s. The Vulkan build carries far less speculative overhead, so it stayed near or above baseline everywhere CUDA cratered.

The practical rule: the faster your model already runs, the more the runtime choice matters. On a fast small model, prefer the Vulkan build or model-free n-gram; the draft-model route earns its keep on slower, bigger models — which is exactly where you wanted the speedup anyway.

What we learned

How this compares to NVIDIA's 15×

It doesn't — and that's the point. NVIDIA's 15× is a throughput number at high concurrency on eight B300s serving a 120B model, where dFlash's block-diffusion drafter and NVFP4 do the heavy lifting. Ours is single-user latency on one consumer card. Different metric, different scale, different drafter.

What transfers is the mechanism and its shape: one expensive pass ratifying several tokens, with a payoff that rises and falls with how predictable the text is — and a sharp dependence on getting the runtime right. dFlash is the datacenter-grade version of a lever you can already pull at home. As stronger open drafters land (EAGLE-3, and dFlash itself), the home version only gets better.

Reproduce it

  1. Grab the latest llama.cpp release — and grab both the CUDA and Vulkan Windows builds so you can compare.
  2. Download any instruct model plus a small same-family draft model (we used Qwen2.5-Coder 7B + 0.5B).
  3. Run llama-server with --spec-type draft-simple -md <draft>, or --spec-type ngram-cache for the zero-download option.
  4. Read timings.predicted_per_second from the server's JSON response, and watch the draft acceptance line in the server log.

Methodology. Single GPU, all model layers on it (--split-mode none -mg 0), context 4096, greedy decoding (temp 0, top-k 1), 256 generated tokens per prompt, a 12-prompt suite spanning code / math / reasoning / summarization / chat. Throughput is the server-reported predicted_per_second (pure generation rate). Every speculative run's output hash matched its baseline, confirming lossless decoding. Numbers vary with model, quant, draft choice, and prompt mix.
