● CPU Threads: More Is Not Always More

More CPU threads made my LLM slower

Setting -t 16 when your CPU has 8 physical cores can halve your tok/s. We sweep thread counts from 1 to 16 and find the knee — and show the hyperthreading penalty in real numbers.

The setup

One small model (Qwen2.5-Coder-3B-Instruct-Q4_K_M), CPU-only inference (-ngl 0 --device none), swept across thread counts: 1, 2, 4, 6, 8, 12, 16 (capped at the logical core count). Both compute (-t) and batch (-tb) threads are set to the same value at each step. Same 12-prompt suite at every count; we report mean generation tok/s.

The story: tok/s rises with threads as cores are added, plateaus around the physical-core count, then often drops as hyperthreads compete for the same memory bandwidth. The optimal setting is almost always at or just below the physical core count — the dotted amber line on the chart.

Reproduce

# drop Qwen2.5-Coder-3B-Instruct-Q4_K_M.gguf into your models dir first
export SPECBENCH_MODELS_DIR=/path/to/models
export SPECBENCH_VULKAN_EXE=/path/to/llama-server
export SPECBENCH_MODEL_BASE=Qwen2.5-Coder-3B-Instruct   # default; change to match your file

python scripts/bench.py

Full guide + how to submit your CPU: CONTRIBUTING.md.

Methodology. CPU-only inference (llama.cpp, -ngl 0 --device none). Thread counts swept 1 → 16 capped at logical core count; -t and -tb set identically. Mean generation tok/s over the shared 12-prompt suite (256-token output, greedy, temp 0, seed 1234). Physical-core count queried from Win32_Processor / /proc/cpuinfo / sysctl hw.physicalcpu. Numbers vary by CPU architecture, BIOS power settings, OS scheduler, memory speed, and llama.cpp version.
