Why thread count matters for CPU inference

Unlike GPU inference — which is bound by fixed GPU parallelism — CPU LLM inference scales with thread count, up to a hard ceiling: your physical core count.

Each forward pass must stream the entire model weight matrix through memory to produce one token. There is almost no weight reuse between tokens. More threads help because more cores can cooperate on that memory transfer — but only while each thread is running on its own dedicated silicon. Hyper-Threading (Intel's SMT) lets each physical core present itself as two logical cores to the OS, but those two logical cores share the same execution units, the same L1/L2 cache, and — critically — the same memory bandwidth. Once every physical core already has a thread, adding more threads via hyperthreading does not give you more bandwidth. It splits the same bandwidth across competing threads while adding OS scheduling overhead.

The practical consequence: tools and OS defaults that set --threads to your logical core count are wrong for LLM inference. Your physical core count (or slightly above) is almost always the sweet spot.

How it works — physical cores vs hyperthreads

The pattern below plays out on any CPU thread sweep: throughput climbs steeply while physical cores are being filled — real bandwidth scaling — then flattens or dips the moment you cross into hyperthread territory, where threads compete for resources they already share.

Results

The curve tells the expected story — with one small twist. Throughput climbs sharply from 1 to 2 threads (5.64 → 9.15 tok/s, +62%), keeps rising through 4 threads (11.76 tok/s), then does something interesting: 6 threads — exactly the physical core count — is marginally slower than 4 (11.57 vs 11.76 tok/s). This is a common quirk when all six cores are simultaneously contending for the memory controller at the same pressure point. Then 8 threads climbs to the actual measured peak (13.0 tok/s): a modest number of hyperthreads can help hide per-core memory latency, so a small amount of HT is net-positive. Finally, 12 threads — all logical cores fully double-booked — drops back to 12.69 tok/s, a 2.4% regression from 8. The "use all cores" default is measurably wrong.

What we learned

The physical core count is the inflection point, not the exact peak

On the i7-8700, the measured optimum is 8 threads, two above the 6 physical cores. A small amount of hyperthreading is net-positive because it lets cores hide memory-access latency during stalls. But there is a cliff: beyond 8, adding more threads into an already-saturated memory bus causes scheduling overhead to outweigh any latency-hiding benefit, and tok/s falls. The rule of thumb is not "stop at physical cores" but "start at physical cores and try +2 — stop well before logical cores."

The biggest gains are entirely inside the physical-core zone

Going from 1 thread to 6 (physical core count) takes throughput from 5.64 to 11.57 tok/s — a 2.05× gain. The best you can extract beyond that, all the way to the measured peak at 8 threads, is another 12% to hit 13.0 tok/s. Everything interesting happens before the hyperthread boundary; everything after is marginal or negative.

Using all logical cores is reliably sub-optimal for this workload

Running at 12 threads (the OS default and many tool defaults) is slower than 8 threads on this hardware — consistently, not just on one prompt. The gap is small (2.4%) but it always goes the wrong direction. There is no scenario in which 12 threads beats 8 here. This matters most on machines where the logical core count is significantly higher than physical (e.g. 8P/16L, 12P/24L) — the penalty scales with how far "use everything" overshoots the sweet spot.

Practical takeaway: set --threads to your physical core count as a starting baseline, then try +2 to see if a small HT bonus materialises on your CPU. Never use the logical core count as your default. On an i7-8700 (6 physical / 12 logical), the measured optimum is 8 threads. On most modern desktop CPUs the answer will be physical cores ± 2.
