The VRAM cliff: what actually happens when your model doesn't fit

Each layer left on the CPU drags throughput down a cliff. We sweep -ngl (GPU-offloaded layers) from 0 to 99 on a fixed model and record tok/s and VRAM at every step — showing the night-and-day gap between full GPU offload and partial CPU spill.

The setup

One model (Qwen2.5-Coder-14B Q4_K_M — large enough that low -ngl values force significant CPU offload), run with the same 12-prompt suite at each layer count. -ngl 0 means every layer runs on CPU; -ngl 99 means all layers fit on GPU. The interesting region is in between: how fast does the cliff drop, and is there a sweet spot where a few GPU layers buy most of the speed?

Reproduce

Place Qwen2.5-Coder-14B-Instruct-Q4_K_M.gguf (or set SPECBENCH_MODEL_BASE) in your models dir (SPECBENCH_MODELS_DIR), then:

python scripts/bench.py --backend cuda --gpu 0
python scripts/aggregate.py && python scripts/inject.py

Full guide + how to submit your hardware: CONTRIBUTING.md.

Methodology. Fixed model and quant (Q4_K_M), greedy (temp 0), 256 tokens, shared 12-prompt suite for tok/s. VRAM = nvidia-smi used-memory delta on model load (approximate; run on an otherwise-idle GPU). Sweep covers -ngl 0, 8, 16, 24, 32, 40, 99. Numbers vary with model, GPU, and llama.cpp version.
