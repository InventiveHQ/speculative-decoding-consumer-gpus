How small a model can actually solve problems?

Tiny models are fast and cheap — but somewhere below 7 billion parameters, they stop being able to reason. We swept Qwen2.5-Coder from 0.5B to 14B on an RTX 5060 Ti, measured where pass-rate collapses, and found the quality knee is sharper than expected.

Why model size matters

Everyone wants the smallest, fastest model that is still actually useful. Smaller means faster inference, lower VRAM requirements, cheaper hardware, and shorter waits. But capability does not scale linearly with size. There is a threshold: below it, a model is blazing fast but cannot reliably solve problems that require chaining steps; above it, it slows down but starts to get things right.

The tension is stark in this sweep. A 0.5B model runs at 381 tok/s on an RTX 5060 Ti. The 14B model runs at 43 tok/s — nearly nine times slower — and occupies 17× more disk space. Whether that cost buys anything worth paying for is the whole question.

How it works — and how we measured

We ran five Qwen2.5-Coder-Instruct checkpoints (0.5B through 14B) through the same harness on the same hardware, collecting two measurements per model:

  • Generation speed (tok/s) — from the shared 12-prompt suite: greedy decoding, 256 output tokens, server-reported predicted_per_second.
  • Pass-rate — a graded set of 10 multi-step math and reasoning problems with known integer answers. Each response is graded by exact match on the last integer in the reply. No partial credit.
What the pass-rate measures — and what it does not. This is a reasoning proxy, not executed code. Running untrusted model-generated code safely inside a benchmark harness is non-trivial, so we approximate with a math and word-problem set. The 10 questions require chaining two to four arithmetic or logical steps — tracking diminishing inventory, computing compound rates, solving simultaneous equations. A model that can chain reasoning steps passes; one that cannot, fails. The sample is small (10 questions), so treat individual scores as signals rather than precise measurements. The 0–1/10 pattern across three consecutive model sizes is consistent enough to be a real signal, not noise.

The setup

Reference test rig. Every “reference” number on this page was measured on one machine — an Alienware Aurora R7:
  • CPU — Intel Core i7-8700 (6 cores / 12 threads, Coffee Lake)
  • Motherboard — Alienware 0VDT73, Intel 300-series — PCIe 3.0 root complex
  • RAM — 64 GB DDR4-2400 (4 × 16 GB, dual-channel)
  • GPU — NVIDIA RTX 5060 Ti 16 GB (Blackwell), running at PCIe 3.0 ×8 · GTX 1080 Ti 11 GB (Pascal) on the multi-GPU experiments
  • Storage — models on a SATA SSD volume (Samsung 860 EVO M.2 + SanDisk) — SATA, not NVMe
  • OS — Windows Server 2025 (build 26100)
In-VRAM throughput is GPU-memory-bandwidth bound, so bus / RAM / SSD speed barely move it — but they shape the offload and CPU results. Running this yourself? scripts/make_submission.py auto-captures your rig (CPU, RAM, GPU + PCIe link, board, storage) into the submission.

Results

The numbers tell a sharp story. At 0.5B, generation flies at 381 tok/s — but the model passes just 1 of 10 reasoning problems, most likely a lucky guess on a 10-question set. The 1.5B model is 3× larger and scores 0 of 10. The 3B model runs at 157 tok/s — still four times the speed of the 14B — and also manages just 1 of 10. Then something changes at 7B: 4 of 10 correct, the first size where the model reliably follows multi-step chains. The 14B reaches 6 of 10 at 43 tok/s, occupying just under 9 GB of VRAM.

The cost of climbing the pass-rate curve is steep. Speed falls from 381 tok/s to 43 tok/s — an 8.9× penalty. File size grows from 531 MB to 8,988 MB — 17× larger. The jump from 7B to 14B alone costs 2× in speed and doubles the VRAM footprint (4,457 MB to 9,179 MB).

On the 1.5B dip to 0/10. It looks worse than 0.5B but with only 10 questions the signal is noisy — the 0.5B's single correct answer may itself be lucky. The consistent signal is that all three models below 7B cluster at 0–1/10 regardless of ordering. None can chain steps reliably.

What we learned

Models at or below 3B cannot do multi-step reasoning

Every model under 7B scored 0 or 1 out of 10. The graded set is not exotic — questions like "a store sold 3/5 of 120 apples in the morning and 25 more in the afternoon; how many remain?" require tracking two steps. Sub-7B models fail on these consistently, not occasionally. Speed does not compensate: the 0.5B runs at 381 tok/s and still cannot get past 1/10. Fast wrong answers are still wrong.

The knee is at 7B — and it is a cliff, not a slope

The jump from 3B (1/10) to 7B (4/10) is the largest qualitative change in the sweep. At 7B, the model tracks intermediate state through a chain of reasoning steps — something the smaller models fail to do consistently. This is not "a little better"; it is the difference between a model that basically cannot reason and one that actually solves problems. If you need multi-step reasoning at all, 7B is the floor, not an option among equals.

Even 14B isn't perfect — and this is a small proxy

The 14B model scores 6 of 10 — the best result, but it still fails on 4 questions. Some failures may be genuinely hard problems; some may be prompt sensitivity on a small set. The 10-question graded set is a signal, not a full benchmark. A real code-execution sandbox with a larger, diverse suite would give a more complete picture. That said, this experiment answers the core question cleanly: models below 7B cannot reliably chain reasoning steps, and the proxy is consistent enough to trust that finding.

Speed and size costs are steep and non-linear

From 0.5B to 14B, speed falls 8.9× and file size grows 17×. The non-linearity matters: the 3B model is 4× the file size of 0.5B but no better on pass-rate. The 14B is 2× the file and VRAM of the 7B for a 50% improvement in pass-rate (4/10 to 6/10). There is no cheap step between 3B and 7B — you have to commit to the full size jump to gain anything.

The practical takeaway

For tasks requiring multi-step reasoning — math, logic, planning, code with non-trivial control flow — do not rely on anything below 7B. The speed advantage of smaller models is real, but you are trading it for a model that cannot chain steps reliably. The 7B is the minimum viable floor: it is genuinely slower (87 tok/s vs 381), but it actually solves problems. The 14B buys additional reliability (6/10 vs 4/10) at another 2× speed penalty; whether that trade-off is worth it depends on your latency tolerance and VRAM budget.

Sub-3B models are not useless — they are fast and capable for single-step tasks: autocomplete, short lookups, simple code completion, summarization where precision matters less. Just do not ask them to reason through a problem that requires more than one step.

Reproduce

Download the Qwen2.5-Coder-Instruct GGUF files into your models dir (SPECBENCH_MODELS_DIR), then:

# run the model-size sweep — missing models are skipped automatically
python scripts/bench.py --backend cuda --gpu 0
python scripts/aggregate.py  # writes results/data.json
python scripts/inject.py     # bakes data into site/index.html

Full guide and how to submit your hardware results: CONTRIBUTING.md.

Methodology. Greedy decoding (temp 0), 256 generated tokens, shared 12-prompt suite for tok/s; a 10-question graded math and reasoning set for pass-rate. VRAM is nvidia-smi used-memory delta on model load (approximate; run on an otherwise-idle GPU). The graded set uses exact match on the last integer in the model reply — no partial credit, no code execution. This is a multi-step reasoning proxy; a code-execution sandbox is a planned future addition. Numbers vary with GPU, llama.cpp version, and quantization.
