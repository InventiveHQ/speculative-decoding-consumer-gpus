Why draft size matters

In experiment #1 we established the core mechanism and its most important finding: speculative decoding speedup is gated by acceptance rate, but that rate has to be high enough to amortize the overhead of running the draft model in the first place. A fast target model can be slow to speculate on not because the draft guesses badly, but because the draft itself costs too much per step relative to what it saves. That is the overhead-vs-acceptance tradeoff, and it is the whole story.

Experiment #1 used a single fixed 0.5B draft against varying target sizes. This experiment flips that: the target is fixed (Qwen2.5-Coder-14B, baseline 42.45 tok/s on CUDA) and we vary only the draft model size — 0.5B, 1.5B, 3B. The question is whether a bigger, smarter draft can close the acceptance gap enough to overcome its higher per-step cost. The answer turns out to be no. Bigger drafts accept more tokens, but the cost rises faster than the acceptance benefit, and the net speedup falls with every size increase.

The size–accuracy tradeoff, visualized

Each draft size creates a different balance between how often the target agrees with the draft (acceptance) and how expensive each draft step is. The setup

Reference test rig. Every “reference” number on this page was measured on one machine — an Alienware Aurora R7:
  • CPU — Intel Core i7-8700 (6 cores / 12 threads, Coffee Lake)
  • Motherboard — Alienware 0VDT73, Intel 300-series — PCIe 3.0 root complex
  • RAM — 64 GB DDR4-2400 (4 × 16 GB, dual-channel)
  • GPU — NVIDIA RTX 5060 Ti 16 GB (Blackwell), running at PCIe 3.0 ×8 · GTX 1080 Ti 11 GB (Pascal) on the multi-GPU experiments
  • Storage — models on a SATA SSD volume (Samsung 860 EVO M.2 + SanDisk) — SATA, not NVMe
  • OS — Windows Server 2025 (build 26100)
In-VRAM throughput is GPU-memory-bandwidth bound, so bus / RAM / SSD speed barely move it — but they shape the offload and CPU results. Running this yourself? scripts/make_submission.py auto-captures your rig (CPU, RAM, GPU + PCIe link, board, storage) into the submission.

Speculative decoding uses a cheap "draft" model to propose tokens that the full target model verifies in one batched pass. In experiment #1 we showed that draft overhead has to be amortized: the speedup only shows up when the acceptance rate is high enough to justify the extra draft cost. This experiment extends that thesis by asking — how much does draft model size matter?

The target is fixed: Qwen2.5-Coder-14B-Instruct-Q4_K_M, CUDA, GPU 0. We first measure a plain baseline (no speculation), then sweep three draft sizes: 0.5B (Q8_0), 1.5B (Q4_K_M), and 3B (Q4_K_M). All drafts use --spec-type draft-simple --spec-draft-n-max 5 -ngld 999 — 5 tokens drafted per round, draft fully on GPU. Speedup is draft tok/s divided by baseline tok/s; above 1.0× means faster than no speculation.

The prediction: a bigger draft accepts more tokens per round (higher acceptance rate) but also burns more VRAM and adds more latency per draft step. The sweet spot likely sits somewhere in the middle.

Results

Reading the numbers: the 0.5B draft reaches 58.25 tok/s (1.37×), with 26% acceptance. The 1.5B achieves 56.01 tok/s (1.32×) at 40% acceptance — noticeably more tokens accepted per round, but the larger model erodes the gain. The 3B hits an impressive 87% acceptance rate, guessing nearly every token correctly, yet produces only 52.91 tok/s (1.25×) — the closest of the three to the 42.45 tok/s baseline. The acceptance–overhead curve never crosses: as draft size grows, the per-step cost rises faster than the acceptance benefit, and each size increment trades net speedup for better guessing accuracy. The sweet spot is at the small end.

The 3B draft agrees with the target 87% of the time — nearly every token — and it is still the slowest of the three. That is the overhead story from experiment #1 made concrete: acceptance rate is necessary but not sufficient. The draft step itself must also be cheap, and a 3B model on the same 16 GB pool as a 14B target is not cheap enough to overcome its accuracy advantage.

What we learned

Acceptance rises steeply with draft size

Going from 0.5B to 3B — a 6× increase in model size — takes acceptance from 26% to 87%. A bigger same-family draft really does predict the target's tokens far more reliably; the capacity difference is real and large. The gain is also nonlinear: 0.5B → 1.5B adds 14 percentage points, while 1.5B → 3B adds another 47. Most of the acceptance benefit is concentrated at the top end of the size range.

Net speedup falls with every size increase

1.37× → 1.32× → 1.25×. Each draft size step costs roughly 0.05–0.07× of net speedup. The 3B's acceptance rate is remarkable — but the model is 3.6× larger than the 0.5B and shares the same GPU memory bandwidth. Those extra weights take time to move through the hardware on every draft cycle, and that overhead compounds regardless of how well the guess turns out.

The 0.5B is the sweet spot on this setup

Not because it is the best guesser — it is clearly not — but because it is cheap enough that even 26% acceptance translates to meaningful leverage on the 14B target. This is the overhead amortization principle from experiment #1 in miniature: speculative decoding earns its keep when the target is slow and the draft is fast. A 14B baseline at 42.45 tok/s is slow enough that a 531 MB draft model is very cheap by comparison, and that ratio is what determines the winner.

VRAM is not the binding constraint

Even the 3B draft only adds ~1,475 MB over the 0.5B configuration (9,769 MB vs 11,244 MB total VRAM used). Fitting a larger draft model is not the problem. The problem is that even a modest VRAM footprint comes with compute cost per draft step, and that cost is what erodes the speedup — not memory capacity.

The practical takeaway

Pick the smallest draft model that keeps acceptance in a productive range. For a same-family draft (Qwen-Coder 0.5B drafting for Qwen-Coder 14B), the 0.5B is already good enough — 26% acceptance on a 42.45 tok/s target translates to 1.37×. Scaling up to 1.5B or 3B costs you speedup without improving wall-clock time. If you change target families, task types, or hardware, re-measure: acceptance is both model-dependent and task-dependent, and the crossover point shifts. The principle from experiment #1 holds: the draft must be cheap and accurate — and cheap matters more than you would think.
