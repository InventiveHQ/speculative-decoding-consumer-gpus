Why this matters — they're all llama.cpp underneath

If you've installed Ollama or LM Studio to run a local model, you're already running llama.cpp — both tools ship it as their inference engine. What you're actually choosing between is which packaging sits between you and the raw binary: a REST server with no extras (llama.cpp itself), a GUI application (LM Studio), or a model-management daemon with its own scheduler and template engine (Ollama). The packaging is the variable. The engine is shared.

That makes the "convenience tax" question interesting and precisely answerable: since the model, quant, and GPU are identical across all three, any speed gap is pure overhead — wrapper version lag, default flags, HTTP and scheduling layers. This experiment puts a number on that tax on real hardware.

How it works — three wrappers, one engine

The setup

Reference test rig. Every “reference” number on this page was measured on one machine — an Alienware Aurora R7:
  • CPU — Intel Core i7-8700 (6 cores / 12 threads, Coffee Lake)
  • Motherboard — Alienware 0VDT73, Intel 300-series — PCIe 3.0 root complex
  • RAM — 64 GB DDR4-2400 (4 × 16 GB, dual-channel)
  • GPU — NVIDIA RTX 5060 Ti 16 GB (Blackwell), running at PCIe 3.0 ×8 · GTX 1080 Ti 11 GB (Pascal) on the multi-GPU experiments
  • Storage — models on a SATA SSD volume (Samsung 860 EVO M.2 + SanDisk) — SATA, not NVMe
  • OS — Windows Server 2025 (build 26100)
In-VRAM throughput is GPU-memory-bandwidth bound, so bus / RAM / SSD speed barely move it — but they shape the offload and CPU results. Running this yourself? scripts/make_submission.py auto-captures your rig (CPU, RAM, GPU + PCIe link, board, storage) into the submission.

All three serve the identical model + quant (Qwen2.5-Coder-7B, Q4) over their OpenAI-compatible endpoints; we hit each with the shared suite, greedy, 256 tokens, and time the response end-to-end. Wall-clock tok/s is the fair metric — it counts whatever overhead each layer adds, not just the model's raw generation rate.

Since all three are the same engine, any gap comes from what the wrapper decides for you: (1) the llama.cpp version it bundles (often behind upstream), (2) default flags (flash-attention, KV-cache type, batch size, context, -ngl), and (3) its own HTTP/templating/scheduling overhead. We can't pin Ollama/LM Studio to a specific llama.cpp build — and that's the point: their bundled version and defaults are what you're measuring.

Results

Reading the numbers: llama.cpp direct hits 77.0 tok/s, LM Studio scores 76.8 tok/s — a gap of 0.2 tok/s, or 0.3% overhead, within measurement noise. LM Studio adds essentially nothing over raw llama.cpp while giving you a full GUI and model manager. Ollama measures 69.1 tok/s — 10.3% slower than the raw binary. That gap likely comes from a combination of its own HTTP scheduling layer, prompt-templating engine, and a bundled llama.cpp that may be behind upstream or use different runtime defaults (flash-attention, batch size, context). Across all five task categories the pattern holds: LM Studio within 2% of llama.cpp everywhere; Ollama consistently 8–14% behind.

Single-device result: only the RTX 5060 Ti was measured here. The 1080 Ti and CPU legs aren't in these numbers because LM Studio's device-pinning isn't scriptable headless — you can't reliably tell it "use GPU 1 only" from a script without GUI interaction. The relative ordering may shift on other hardware, especially if Ollama's bundled llama.cpp version has better support for older or slower devices. Contributions from other devices are welcome (see Reproduce below).

What we learned

LM Studio adds ~0% overhead — it's llama.cpp with a GUI

At 76.8 vs 77.0 tok/s, LM Studio's convenience layer costs you 0.3% throughput. That is within measurement noise on a 12-prompt suite. The GUI, model downloader, and OpenAI-compatible server are effectively free from a performance standpoint — LM Studio does not add a meaningful scheduling or buffering layer on top of llama.cpp's own server path.

Ollama adds ~10% overhead — the packaging is real

At 69.1 tok/s, Ollama runs 7.9 tok/s behind raw llama.cpp — a consistent 10% tax across all task categories. The likely sources: Ollama's own daemon layer (handles model lifecycle, context multiplexing, and request queuing), its prompt-template engine, and potentially a bundled llama.cpp that's behind the upstream release or uses different runtime defaults. None of this is unreasonable engineering for what Ollama promises (zero-config model management, a model hub, multi-model serving) — but it has a measurable cost.

The gap is packaging, not the model or engine

All three runners loaded the same GGUF file and ran the same GPU kernels. The performance difference is purely what each wrapper decides to do on top. Knowing this means you can predict the trajectory: as Ollama ships newer llama.cpp builds and tunes its defaults, the overhead will likely narrow. The engine is not the variable — the wrapper is.

Takeaway: which runner should you pick?

Raw llama.cpp is the ceiling — 77.0 tok/s, no abstractions, full flag control. The right choice if you are benchmarking, scripting, or need maximum throughput. The tradeoff is setup friction: no GUI, no model hub, manual config and flags.

LM Studio is the best of both worlds if you want a GUI: 76.8 tok/s (0.3% behind raw), with a polished model downloader, chat interface, and OpenAI-compatible server. On this hardware the convenience is essentially free from a performance standpoint.

Ollama is the right choice if you value its ecosystem — pull-by-name model management, the Modelfile system, Docker-friendly daemon model, and a large community hub. The 10% throughput cost (69.1 tok/s) is real, but still fast for a 7B model on a consumer GPU. If Ollama's UX saves you setup friction, the trade is likely worth it — and the gap will probably narrow as Ollama ships newer llama.cpp builds.

Reproduce

For each device, pin all three runners to it (the same model + quant), then run the suite tagged with that device. Device pinning differs per runner — e.g. CUDA_VISIBLE_DEVICES=0|1 (or empty for CPU) before ollama serve; flags/env for llama-server; LM Studio's GUI:

# pin each runner to the target device, then:
python scripts/bench.py --device 5060ti
python scripts/bench.py --device 1080ti
python scripts/bench.py --device cpu
python scripts/aggregate.py && python scripts/inject.py

Full per-runner pinning recipes + how to submit your hardware: CONTRIBUTING.md and this experiment's README.

Methodology. Same model/quant per runner, OpenAI /v1/chat/completions, greedy (temp 0), 256 generated tokens, shared 12-prompt suite. Metric = wall-clock completion_tokens / elapsed for cross-runner fairness. Numbers vary with version, GPU, and config.
