They're all llama.cpp underneath — so why is one slower?

LM Studio and Ollama both run on llama.cpp. So this isn't three engines racing — it's raw llama.cpp vs two convenience layers on top, same model and hardware, same 12-prompt suite. We measure wall-clock tokens/sec to quantify what the convenience actually costs you.

The setup

All three serve the identical model + quant (Qwen2.5-Coder-7B, Q4) over their OpenAI-compatible endpoints; we hit each with the shared suite, greedy, 256 tokens, and time the response end-to-end. Wall-clock tok/s is the fair metric — it counts whatever overhead each layer adds, not just the model's raw generation rate.

Since all three are the same engine, any gap comes from what the wrapper decides for you: (1) the llama.cpp version it bundles (often behind upstream), (2) default flags (flash-attention, KV-cache type, batch size, context, -ngl), and (3) its own HTTP/templating/scheduling overhead. We can't pin Ollama/LM Studio to a specific llama.cpp build — and that's the point: their bundled version and defaults are what you're measuring.

Reproduce

Start each runner with the same model, then run the suite:

llama-server -m Qwen2.5-Coder-7B-Instruct-Q4_K_M.gguf -ngl 999 --port 8080 --jinja
ollama serve              # ollama pull qwen2.5-coder:7b
# LM Studio: load the same model, start its local server (:1234)

python scripts/bench.py --out results/results.json
python scripts/aggregate.py && python scripts/inject.py

Full guide + how to submit your hardware: CONTRIBUTING.md.

Methodology. Same model/quant per runner, OpenAI /v1/chat/completions, greedy (temp 0), 256 generated tokens, shared 12-prompt suite. Metric = wall-clock completion_tokens / elapsed for cross-runner fairness. Numbers vary with version, GPU, and config.
