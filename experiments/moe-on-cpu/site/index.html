MoE on CPU: 13B-class answers at 3B speed

A Mixture-of-Experts model carries a huge parameter count but only fires a slice of it per token. On a CPU — where decoding is memory-bandwidth-bound — does it run at the speed of its total size, or its active size? We measured a 30B-A3B MoE against dense 1.5B–8B models on an 8-year-old i7-8700.

The question: do you pay for the parameters you store, or the ones you use?

Dense models read every weight to generate every token. A Mixture-of-Experts (MoE) model is built differently: its layers hold many parallel "expert" sub-networks, and a small router picks just a few to run for each token. A 30B-A3B model stores 30 billion parameters but activates only about 3.3 billion per token — the "A3B" part.

That distinction barely matters on a fast GPU, but it is decisive on a CPU. Single-user CPU generation is memory-bandwidth-bound: the speed ceiling is set by how many weight bytes you stream from RAM for each token. If only the active experts are read, an MoE should decode at roughly the speed of a dense model its active size — while answering like something far bigger. This experiment tests exactly that, on hardware most people already own.

Why not dFlash, or speculative decoding? NVIDIA's dFlash (and EAGLE-3) hit their headline multiples with a trained, target-conditioned draft head running in TensorRT-LLM on Blackwell GPUs at high concurrency — not something that exists in llama.cpp or on a CPU, and there is no published head for this model. And as experiment #1 showed, at batch 1 on a single device a generic draft is overhead-governed and often loses. So the interesting lever here is not a bolt-on decoder trick — it is the MoE architecture itself. That is what we measure.

How MoE routing saves bandwidth

The catch is memory capacity, not speed: all 30B of experts have to fit in RAM, because which ones fire changes token to token. At Q4 that is ~18 GB — fine on a 32 GB+ machine, impossible to keep resident on most 16 GB GPUs. CPU + lots of cheap system RAM is the MoE's natural home.

The setup

Reference test rig. Every “reference” number on this page was measured on one machine — an Alienware Aurora R7:
  • CPU — Intel Core i7-8700 (6 cores / 12 threads, Coffee Lake)
  • Motherboard — Alienware 0VDT73, Intel 300-series — PCIe 3.0 root complex
  • RAM — 64 GB DDR4-2400 (4 × 16 GB, dual-channel)
  • GPU — NVIDIA RTX 5060 Ti 16 GB (Blackwell), running at PCIe 3.0 ×8 · GTX 1080 Ti 11 GB (Pascal) on the multi-GPU experiments
  • Storage — models on a SATA SSD volume (Samsung 860 EVO M.2 + SanDisk) — SATA, not NVMe
  • OS — Windows Server 2025 (build 26100)
This experiment runs entirely on the CPU (-ngl 0), so the lever is DDR4-2400 dual-channel memory bandwidth — DDR5 or more channels should lift the whole curve. Running this yourself? scripts/make_submission.py auto-captures your rig (CPU, RAM speed, board, storage) into the submission.

The bottleneck is RAM, and RAM is cheap

Two machines decode this MoE at very different speeds depending entirely on memory bandwidth: dual-channel DDR4-2400 (this box) sits at the low end; DDR5-6000 or a quad/eight-channel workstation moves the whole curve up. But unlike VRAM, system RAM is cheap and plentiful — putting a 30B-class model within reach of a machine with no usable GPU at all. That is the real unlock: you are no longer gated by a 16 GB VRAM ceiling, only by how many DIMMs you can afford.

This is a quality proxy, on a small set

The 10-question graded set measures multi-step reasoning by exact-answer match — it is a signal, not a full benchmark, and no model code is executed. Treat the quality numbers as "which tier is this model in," not a leaderboard. The speed result, by contrast, is robust: it is a direct consequence of how many bytes the runtime streams per token.

Reproduce

Get a recent llama.cpp build (the MoE arch is new), download the GGUFs into your models dir, then:

# CPU-only sweep — missing models are skipped automatically
python scripts/bench.py --threads 6
python scripts/aggregate.py  # writes results/data.json
python scripts/inject.py     # bakes data into site/index.html

Memory bandwidth is the whole story here, so DDR5 / more channels should shift your curve up — submissions across memory configs are especially valuable. Full guide: CONTRIBUTING.md.

Methodology. CPU only (-ngl 0), 6 threads. Greedy decoding (temp 0), 256 generated tokens, shared 12-prompt suite for tok/s (server predicted_per_second); a 10-question graded math/reasoning set for quality (exact match on the last integer, no partial credit, no code execution). The "dense 30B" reference is an extrapolation of the dense points' bandwidth trend (tok/s ∝ 1/active), shown for scale only. Active/total parameter counts are the published model specs. Numbers vary with CPU, RAM bandwidth, thread count, and llama.cpp version. The MoE is Qwen3-VL-30B-A3B-Instruct; its text decoder is the same A3B MoE used for text-only generation here.
