The context-length tax: what 2K→32K costs you

Everyone cranks -c 32768 "just in case." Reserving that headroom costs ~1.7 GB of VRAM immediately — but not a single tok/s of generation speed. Here's the sweep across five context sizes, on real hardware.

Why context length matters

When you launch llama-server with -c 32768, the runtime allocates a KV-cache sized for the full 32,768 tokens — right now, before your first prompt arrives. The model has no idea how long your actual inputs will be; it reserves the worst case up front.

The common pattern is to set context large "just in case" — a long coding session, a big paste, a multi-turn conversation. But that reservation has a cost. This experiment makes it concrete: same model, same quant, five context sizes from 2K to 32K. What do you actually pay?

How it works

The KV-cache is not dynamically allocated as your conversation grows — it is sized at startup for the declared maximum. You pay for 32K tokens of cache from minute one, whether you use 200 tokens or 32,000.

The setup

Reference test rig. Every “reference” number on this page was measured on one machine — an Alienware Aurora R7:
  • CPU — Intel Core i7-8700 (6 cores / 12 threads, Coffee Lake)
  • Motherboard — Alienware 0VDT73, Intel 300-series — PCIe 3.0 root complex
  • RAM — 64 GB DDR4-2400 (4 × 16 GB, dual-channel)
  • GPU — NVIDIA RTX 5060 Ti 16 GB (Blackwell), running at PCIe 3.0 ×8 · GTX 1080 Ti 11 GB (Pascal) on the multi-GPU experiments
  • Storage — models on a SATA SSD volume (Samsung 860 EVO M.2 + SanDisk) — SATA, not NVMe
  • OS — Windows Server 2025 (build 26100)
In-VRAM throughput is GPU-memory-bandwidth bound, so bus / RAM / SSD speed barely move it — but they shape the offload and CPU results. Running this yourself? scripts/make_submission.py auto-captures your rig (CPU, RAM, GPU + PCIe link, board, storage) into the submission.

One model (Qwen2.5-Coder-7B-Instruct Q4_K_M), one backend (CUDA), five context sizes: 2 K, 4 K, 8 K, 16 K, 32 K. For each we launch a fresh llama-server, let it fully load, then measure:

  • Generation tok/s — the shared 12-prompt suite, averaged.
  • VRAM used — nvidia-smi delta between idle and loaded (the KV-cache reservation is the dominant contributor; it scales with context).
  • TTFT proxy — timings.prompt_ms from a fixed ~200-token prompt (prompt-evaluation latency; rises with context because the attention mask grows).

The model doesn't know how long future inputs will be; it allocates the full KV-cache up front. Every extra kilotoken of declared context is VRAM you pay for immediately — even if your actual prompt is three sentences.

Results

The headline across all five runs: generation tok/s is flat at ~80 regardless of context size — ranging from 77.8 to 81.3 tok/s, well within measurement noise. VRAM, however, climbs steadily from 4,559 MiB at 2K to 6,269 MiB at 32K — a clean +1,710 MiB (+1.7 GB). TTFT barely moves: 85.8 ms at 2K vs 88.8 ms at 32K (+3 ms). The 8K point showed a one-off spike to 92.5 ms, likely measurement noise on a single run. The chart tells the story plainly: the speed line is flat while the VRAM line climbs.

What we learned

Generation speed is bound by model weights, not unfilled context

tok/s held at ~80 across every context size (2K: 80.2, 4K: 81.3, 8K: 77.8, 16K: 80.6, 32K: 80.3). Generation is memory-bandwidth-bound: the bottleneck is reading the weight matrices through the GPU on every forward pass. The unfilled KV-cache slots don't factor in until you actually fill them. Declaring 32K instead of 2K does not slow your output stream at all — the speed you get today is the speed you keep.

VRAM cost is real, linear, and charged immediately

VRAM went from 4,559 MiB at 2K to 6,269 MiB at 32K — a clean +1,710 MiB allocated the moment llama-server starts, before a single token is generated. The relationship is roughly linear with context: each doubling of context adds roughly 300–600 MiB for this model and quant. On a VRAM-tight card, that 1.7 GB could be the difference between fitting a second model, enabling a quantized KV-cache, or not.

TTFT impact is modest — the real cost is memory, not latency

Time-to-first-token increased by only +3 ms from 2K to 32K (85.8 → 88.8 ms). Even the noisy 8K spike (92.5 ms) is only +6.7 ms over the 2K baseline. For almost any interactive workload this is imperceptible. If you are chasing latency, the context-window setting is not the knob to reach for. The story here is VRAM, not milliseconds.

Practical takeaway: Set -c to what you actually need, not "just in case." The performance tax for unused context is almost entirely VRAM — not generation speed. On a 16 GB card, going from 2K to 32K costs ~1.7 GB of headroom you might need for a bigger model, a second model, or KV-cache quantization. If you regularly paste large inputs, pay that cost deliberately — not by habit.

Methodology. Fixed model (Qwen2.5-Coder-7B-Instruct Q4_K_M), greedy (temp 0), 256 max output tokens, shared 12-prompt suite for tok/s. VRAM = nvidia-smi used-memory delta on model load (run on an otherwise-idle GPU for a clean delta). TTFT proxy = timings.prompt_ms reported by llama.cpp for a fixed ~200-token prompt after warmup. Numbers vary with GPU, driver, and llama.cpp version; use the same binary version for apples-to-apples.
