● Context-Length Tax

The context-length tax: what 2K→32K costs you

Declaring a larger context window costs throughput and VRAM even when you don't fill it. Same model, same quant — we sweep 2 048 → 32 768 tokens and measure exactly how much speed and memory you're paying per extra kilotoken of headroom.

The setup

One model (Qwen2.5-Coder-7B-Instruct Q4_K_M), one backend (CUDA), five context sizes: 2 K, 4 K, 8 K, 16 K, 32 K. For each we launch a fresh llama-server, let it fully load, then measure:

  • Generation tok/s — the shared 12-prompt suite, averaged.
  • VRAM used — nvidia-smi delta between idle and loaded (the KV-cache reservation is the key contributor).
  • TTFT proxy — timings.prompt_ms from a fixed ~200-token prompt (prompt-evaluation latency; rises with context because the attention mask grows).

The model doesn't know how long future inputs will be; it allocates the full KV-cache up front. Every extra kilotoken of declared context eats VRAM and slows the prefill — even if your actual prompt is three sentences.

Reproduce

Place Qwen2.5-Coder-7B-Instruct-Q4_K_M.gguf in your models dir, set SPECBENCH_MODELS_DIR, then:

python scripts/bench.py --backend cuda --gpu 0
python scripts/aggregate.py && python scripts/inject.py

Full guide + how to submit your hardware: CONTRIBUTING.md.

Methodology. Fixed model (Qwen2.5-Coder-7B-Instruct Q4_K_M), greedy (temp 0), 256 max output tokens, shared 12-prompt suite for tok/s. VRAM = nvidia-smi used-memory delta on model load (run on an otherwise-idle GPU for a clean delta). TTFT proxy = timings.prompt_ms reported by llama.cpp for a fixed ~200-token prompt after warmup. Numbers vary with GPU, driver, and llama.cpp version; use the same binary version for apples-to-apples.
