● Quantization: Speed vs Quality

How low can you quantize before it breaks?

Quantizing a GGUF model is the single most effective lever for fitting a large model on a small card and generating tokens faster — but it's lossy. We swept Qwen2.5-Coder-7B from Q2 to Q8 on an RTX 5060 Ti and measured exactly where output quality falls off a cliff.

Why quantization is the lever everyone needs to understand

Running Qwen2.5-Coder-7B in its native 16-bit floating-point format requires roughly 14 GB of VRAM — more than most consumer GPUs have. Quantization maps each weight from a 16-bit float to a smaller integer format, trading numerical precision for a massive reduction in file size and VRAM footprint. That smaller footprint means more of the model fits in fast GPU memory, which directly accelerates generation.

The reason smaller means faster is that local LLM inference is almost entirely memory-bandwidth-bound: the GPU spends most of each forward pass streaming the model's weights from VRAM to compute cores. A model that is half the size streams in roughly half the time, so lower-bit quants genuinely go faster — not because they do less computation, but because the data movement is cheaper. Q4_K_M on this 7B model uses 4,673 MB of VRAM vs 7,669 MB at Q8_0, and generates tokens ~1.6× faster.

The catch is that quantization is lossy and irreversible. At high bit-depths (Q6, Q8) the rounding error is small and practically invisible. But push to Q3 or Q2 and the numerical errors compound across thousands of layers until the generated text diverges meaningfully from what the full-precision model would have said. This experiment measures exactly where that happens.

How quantization works — and where fidelity breaks

Each weight in the model is a floating-point number. Quantization groups weights into blocks and maps the values in each block to a small set of representable integers: 8 bits gives 256 levels of precision, 4 bits gives 16, 2 bits gives only 4. The fewer the levels, the coarser the rounding, the larger the accumulated error in outputs. GGUF's K-quant family (Q3_K_M, Q4_K_M, Q5_K_M…) stores a per-block scaling factor that recovers some quality — but the fundamental limit is always the bit depth. Below Q4, there simply are not enough levels to represent the model's weight distribution cleanly.

The similarity score measures how much quantization changed the model's greedy token choices vs the Q8_0 reference — it is a drift proxy, not an absolute correctness guarantee. A 57%-similar model might still be perfectly adequate for your use case; a 90%-similar one might still get a critical answer wrong. Use it as a relative signal: a large drop tells you the quant is substantially altering outputs, not just slightly rounding them.

The setup

Reference test rig. Every “reference” number on this page was measured on one machine — an Alienware Aurora R7:
  • CPU — Intel Core i7-8700 (6 cores / 12 threads, Coffee Lake)
  • Motherboard — Alienware 0VDT73, Intel 300-series — PCIe 3.0 root complex
  • RAM — 64 GB DDR4-2400 (4 × 16 GB, dual-channel)
  • GPU — NVIDIA RTX 5060 Ti 16 GB (Blackwell), running at PCIe 3.0 ×8 · GTX 1080 Ti 11 GB (Pascal) on the multi-GPU experiments
  • Storage — models on a SATA SSD volume (Samsung 860 EVO M.2 + SanDisk) — SATA, not NVMe
  • OS — Windows Server 2025 (build 26100)
In-VRAM throughput is GPU-memory-bandwidth bound, so bus / RAM / SSD speed barely move it — but they shape the offload and CPU results. Running this yourself? scripts/make_submission.py auto-captures your rig (CPU, RAM, GPU + PCIe link, board, storage) into the submission.

One model — Qwen2.5-Coder-7B-Instruct — run through each GGUF quant level from Q2_K up to Q8_0 on an RTX 5060 Ti (16 GB, Blackwell) using llama.cpp's CUDA backend. Per quant we record generation speed, VRAM footprint, file size, and two quality signals: a graded math pass-rate over 8 multi-step problems with exact integer answers, and output similarity vs Q8_0 (mean difflib sequence-match ratio over the shared 12-prompt suite).

Results

The speed chart shows fewer bits trending faster. Q2_K peaks at 91.2 tok/s, Q8_0 bottoms at 49.6 tok/s — a 1.84× range. But the trend is not a clean monotone: Q3_K_M (73.0 tok/s) is actually slower than Q4_K_M (79.0 tok/s) despite having fewer bits. This is a K-quant structure quirk — Q3_K_M's dequantization layout can be less cache-friendly than Q4_K_M's, so the expected "fewer bits = faster" rule has local exceptions. Frame the speed trend as a general gradient, not a strict line; small differences between adjacent quant levels are noise.

The quality chart is where the real story is. The math pass-rate is nearly flat — 3/8 across Q3_K_M through Q8_0, dropping one point to 2/8 only at Q2_K. This graded set is intentionally hard, and a 7B coder model at these quant levels solves around 3 of 8 regardless of quantization depth — not because quant doesn't matter, but because this particular test is not sensitive enough to separate the levels. The output-similarity line tells a much clearer story: it falls from 100% at Q8_0 to 79.7% at Q6_K, to 56.6% at Q4_K_M, then drops steeply to 40.9% at Q3_K_M and all the way down to 21.5% at Q2_K. At that point, nearly 4 in 5 output tokens differ from what Q8_0 would have generated — this is not minor rounding noise; the model is qualitatively different.

What we learned

The fidelity cliff is real and steep below Q4

From Q8_0 down to Q4_K_M, similarity drops from 100% to 56.6% — a gradual 43-point slide over four quant levels. Then Q4_K_M to Q3_K_M loses another 16 points in one step (56.6% → 40.9%), and Q3_K_M to Q2_K drops 19 more (40.9% → 21.5%). The cliff is not a metaphor: the loss rate roughly doubles below Q4. For tasks that require factual accuracy, code correctness, or coherent multi-step reasoning, Q2 and Q3 should be treated as genuinely lossy formats — not just slightly compressed versions of the same model.

The practical knee is Q4–Q5

Q4_K_M cuts the file from 8.1 GB to 4.7 GB (42% smaller) and VRAM from 7,669 MB to 4,673 MB, while output similarity sits at 56.6% — well above the cliff. Q5_K_M pushes similarity to 68.9% at 5.4 GB and 5,333 MB; that extra ~700 MB buys a meaningful fidelity improvement. The Q5→Q6 step (68.9% → 79.7%) costs another ~0.9 GB for a solid quality gain, and is worth it when VRAM headroom allows. But the biggest bang-per-gigabyte is in the Q4→Q5 range. Below Q4 you are buying speed and size at a disproportionate quality cost.

Speed favours lower bits, but not in a straight line

Broadly, fewer bits streams through memory faster — the model is smaller, and inference is bandwidth-bound. The range across the full sweep is 91.2 tok/s (Q2_K) vs 49.6 tok/s (Q8_0), a genuine 1.84× at the extremes. Within the middle range, do not over-read small gaps: Q3_K_M at 73.0 tok/s trails Q4_K_M at 79.0 tok/s, an artifact of K-quant dequantization overhead rather than a principle. Speed trends are real at scale; adjacent-level comparisons can reverse.

The math pass-rate is not a sensitive quality signal for this sweep

The 8-question graded math set is hard (multi-step word problems, exact integer extraction), and a 7B instruct model scores 3/8 across almost all quantization levels. Q2_K drops to 2/8 — a data point, not a verdict. The flat line does not mean quant doesn't affect correctness; it means this particular probe isn't fine-grained enough to separate the levels. The output-similarity slope is the primary quality signal in this experiment. Perplexity and KL-divergence vs FP16 would be more rigorous; those are reserved for a future pass.

"Similarity" measures drift from Q8_0, not absolute quality

Output similarity tells you how much quantization changed the model's greedy completions relative to Q8_0 — it is a drift proxy, not a correctness certificate. Two important caveats: (1) Q8_0 itself introduces some quantization error vs FP16; it is the highest-precision quant in this sweep, not ground truth. (2) A lower-similarity quant might still produce correct code or right answers on your task — or a high-similarity quant might still fail on edge cases. Run your own prompts on Q4 and Q5 before committing to a choice for production use.

Takeaway: Q4_K_M or Q5_K_M for most use cases

Q4_K_M is the default recommendation for consumer GPU use: 42% VRAM savings vs Q8_0, ~1.6× faster generation, and output similarity of 57% — below the quality cliff, but above the collapse zone. Q5_K_M is worth choosing when VRAM allows and fidelity matters more than the last bit of speed — similarity climbs to 69% at modest cost. Q6_K reaches 80% similarity and is reasonable if you have 6 GB to spare on the model. Avoid Q2_K and Q3_K_M unless you have explicitly tested your target task and found the output acceptable — at 21% and 41% similarity respectively, these are qualitatively different models, not mildly lossy compressions. If Q4_K_M doesn't fit your VRAM budget, try Q3_K_M only after benchmarking; do not assume it will behave like Q4.

Reproduce

Download the quants into your models dir (e.g. bartowski's Qwen2.5-Coder-7B GGUF repo: Q2_K, Q3_K_M, Q4_K_M, Q5_K_M, Q6_K, Q8_0), then:

# run the quant sweep — missing quants are skipped automatically
python scripts/bench.py --backend cuda --gpu 0
python scripts/aggregate.py  # → results/data.json
python scripts/inject.py     # bakes numbers into site/index.html

Run once per device if you want the speed/quality tradeoff on your own hardware. Best on an otherwise-idle GPU so the VRAM delta is clean. Full guide and how to submit your results: CONTRIBUTING.md.

Methodology. Same model (Qwen2.5-Coder-7B-Instruct), greedy decoding (temp 0), 256 tokens, shared 12-prompt suite for tok/s and output-drift measurement; an 8-question graded math set (multi-step word problems, exact integer match) for pass-rate. VRAM = nvidia-smi used-memory delta on model load (approximate; run on an otherwise-idle GPU). Output similarity = mean difflib sequence-match ratio of each quant's greedy completions vs Q8_0. Speed trend is broadly "fewer bits = faster" but is not perfectly monotonic — Q3_K_M (73.0 tok/s) trails Q4_K_M (79.0 tok/s) due to K-quant dequantization structure. Numbers vary with model, GPU, driver, and llama.cpp version.
