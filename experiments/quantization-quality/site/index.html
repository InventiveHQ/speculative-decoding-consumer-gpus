● Quantization: Speed vs Quality

How low can you quantize before it breaks?

Q2 to Q8 on the same model. Quantizing buys you speed and VRAM — but somewhere down there, quality falls off a cliff. We measure tok/s, VRAM, and two quality signals at each level to find exactly where.

The setup

One model (Qwen2.5-Coder-7B), the same prompts, run through each GGUF quant from Q2_K up to Q8_0. Per level we record generation tok/s, VRAM used, and file size — plus two quality signals: a graded math pass-rate (does it still get the answer right?) and output similarity — how close each quant's greedy output stays to the highest-precision quant's (the closer to 100%, the less the quantization changed what the model says).

Reproduce

Download the quants into your models dir (e.g. bartowski's Qwen2.5-Coder-7B GGUF: Q2_K, Q3_K_M, Q4_K_M, Q5_K_M, Q6_K, Q8_0), then:

python scripts/bench.py --backend cuda --gpu 0   # missing quants are skipped
python scripts/aggregate.py && python scripts/inject.py

Full guide + how to submit your hardware: CONTRIBUTING.md.

Methodology. Same model, greedy (temp 0), 256 tokens, shared 12-prompt suite for tok/s + output-drift; an 8-question graded math set for pass-rate. VRAM = nvidia-smi used-memory delta on model load (approximate; run on an otherwise-idle GPU). Output similarity = mean difflib ratio of each quant's greedy output vs the highest-precision quant present. Numbers vary with model, GPU, and llama.cpp version.
