Tokens per watt: I capped my GPU to 70% power and barely lost speed

Decoding is memory-bandwidth-bound, not compute-bound. That means you can often slash the GPU's power limit and barely lose a token per second — while running cooler, quieter, and more efficiently.

Why power matters for local inference

Running a 7B model all day costs real electricity, heat, and fan noise. A GPU pulling 173 W continuously adds up fast. The obvious fix is a power cap. The obvious worry is that you'll sacrifice speed. Single-user LLM decoding is memory-bandwidth-bound: at batch size 1 the GPU mostly waits for weights to stream from VRAM. The arithmetic units sit idle most of the time. Extra compute clocks — bought with extra watts — don't move the needle on throughput. Cutting power throttles clocks, but clocks are not the bottleneck.

This experiment measures exactly how much speed you lose, and how much efficiency you gain, as the power cap steps from the 5060 Ti's default (180 W) down to its hardware floor (150 W).

How it works: the bandwidth-bound argument

Each token forward pass reads the full model weight matrix from VRAM — several gigabytes streamed sequentially. The arithmetic to apply those weights finishes in microseconds; waiting for the next weight tile from VRAM takes milliseconds. Power throttling reduces clock frequency, which reduces how fast the math runs — but math is not what you are waiting for. Speed stays flat until you throttle so hard that the memory controller itself slows, which does not happen in the 150–180 W range on this card.

The RTX 5060 Ti's driver-enforced minimum power limit is 150 W — it will not go lower. This experiment tested only three points: 150 W, 162 W, and 180 W (default). Cards with lower hardware floors (e.g., older 30xx/40xx series) can be capped further; the pattern is the same, but the efficiency gains scale wider.

The setup

One model (Qwen2.5-Coder-7B-Instruct Q4_K_M), one CUDA GPU, the shared 12-prompt suite. We query the GPU's default power limit via nvidia-smi, then sweep down through 100 → 90 → 80 → 70 → 60 → 50% of that limit. At each step we set the cap, launch the server fresh, run the full prompt suite measuring generation tok/s, and sample the actual power draw in a background thread throughout — then compute tokens per watt.

The intuition: at small batch sizes (which is every local-LLM run), the GPU sits mostly idle waiting for VRAM reads. The arithmetic flops are done long before the next weight tile arrives. Cutting power mostly throttles compute clocks, which aren't the bottleneck — so speed barely budges while efficiency climbs.

Results

The data tells a clean story. Cutting the power cap from 180 W to 150 W moves measured draw from 173 W to 147 W — a real 26 W reduction. Throughput slips from 81.2 to 80.4 tok/s, a difference of less than 1%. Efficiency climbs from 0.470 to 0.546 tok/s/W, a +16% improvement. The middle point (162 W cap, 159 W draw, 81.4 tok/s, 0.511 tok/s/W) confirms the trend is monotone: every watt removed improves efficiency without meaningfully denting speed. This is the memory-bandwidth-bound regime in action — the work gets done at the same rate, it just costs fewer joules.

What we learned

Almost no speed lost — ~1% for 25 fewer watts

Going from 180 W to 150 W costs 0.8 tok/s (81.2 → 80.4). That rounds to zero in any practical context. The GPU's compute clocks throttle, but compute was never the bottleneck — VRAM bandwidth is. Weight tensors stream at the same rate regardless of how fast the arithmetic units run once they receive them.

+16% efficiency is real and free

0.470 → 0.546 tok/s/W is a genuine 16% improvement in useful work per joule. Running an inference server around the clock at 150 W instead of 180 W saves roughly 250 Wh per day — about $0.03/day at average US electricity rates, or roughly $11/year — plus meaningful heat and noise reduction. The efficiency gain costs nothing except a one-time nvidia-smi -pl 150 call before starting the server.

Why bandwidth-bound workloads ignore watts

At batch size 1, the GPU processes one token at a time. Each token pass reads the full model weight matrix from VRAM — several gigabytes of sequential reads. The arithmetic to apply those weights finishes in microseconds; waiting for the next VRAM tile takes milliseconds. Power throttling reduces clock speed, which reduces how fast the arithmetic finishes — but that does not matter, because the limiting step is always the memory bus. Speed stays flat until you throttle so hard that the memory controller itself clocks down, which for the 5060 Ti does not happen between 150 W and 180 W.

The 150 W floor caveat

The RTX 5060 Ti enforces a minimum power limit of 150 W. The driver refuses lower values. Only three points were measurable in this sweep. A card with a lower hardware floor would expose a wider range and potentially larger efficiency gains; the bandwidth-bound shape would be identical.

Takeaway: cap it

If you run local LLM inference on an NVIDIA GPU, set the power limit to its hardware floor. On an RTX 5060 Ti that is nvidia-smi -pl 150. You will lose less than 1% of your throughput and gain ~16% efficiency, a cooler card, and quieter fans. The command requires elevated privileges (Windows: Administrator prompt; Linux: sudo) and does not survive a reboot — add it to a startup script or run it before your inference server starts.

The principle generalises beyond this card: any single-user, batch-size-1 decode workload is memory-bandwidth-bound. Power caps hurt throughput only when compute is the bottleneck — and at batch 1 on consumer hardware it is not. The flat-then-cliff efficiency curve is a consistent property of the regime, not of this specific GPU.

Reproduce

Requires an NVIDIA GPU and llama-server. Running this modifies GPU power limits and requires administrator/elevated privileges (Windows: Administrator prompt; Linux: sudo). The script always restores the default limit in a finally block.

python scripts/bench.py --gpu 0
python scripts/aggregate.py && python scripts/inject.py

Set SPECBENCH_MODELS_DIR to your models folder, SPECBENCH_CUDA_EXE to your llama-server binary, and optionally SPECBENCH_MODEL_BASE (default Qwen2.5-Coder-7B-Instruct). The script expects <MODEL_BASE>-Q4_K_M.gguf.

Full guide + how to submit your hardware: CONTRIBUTING.md.

Methodology. Fixed model (Qwen2.5-Coder-7B-Instruct Q4_K_M), greedy (temp 0), 256 tokens, shared 12-prompt suite. Power draw sampled via nvidia-smi --query-gpu=power.draw every 0.5 s in a background thread throughout each suite run; average taken across all samples. tok/s/W = mean tok/s / mean power draw. VRAM is already resident — power limit changes affect clocks, not memory. Numbers vary with GPU model, driver, and thermal state.
