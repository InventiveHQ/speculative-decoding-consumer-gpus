● Flash Attention: Speed & VRAM Tradeoff

Flash attention: is -fa actually free?

The -fa flag in llama.cpp promises lower KV-cache memory and sometimes faster generation. But does it deliver on a consumer GPU? We sweep three context lengths — 4K, 16K, 32K — with flash-attention off and on, and measure tok/s and VRAM at each.

The setup

One model, one quant (Qwen2.5-Coder-7B-Instruct Q4_K_M), CUDA backend. The grid: context lengths 4096 / 16384 / 32768 × flash-attention off / on. For each cell we run the shared 12-prompt suite and record generation tok/s, plus VRAM used via nvidia-smi delta on model load.

The story: -fa cuts KV-cache memory — giving more headroom at long context — and can speed up generation by doing the attention computation in a fused kernel. At short context the gains may be negligible; at 32K they should be real. We measure where it helps vs where it's a wash.

Reproduce

Place Qwen2.5-Coder-7B-Instruct-Q4_K_M.gguf in your models dir, then:

python scripts/bench.py --gpu 0          # sweeps 6 cells (3 ctx x fa off/on)
python scripts/aggregate.py              # -> results/data.json
python scripts/inject.py                 # bakes into site/index.html

Full guide + how to submit your hardware: CONTRIBUTING.md.

Methodology. Fixed model (Qwen2.5-Coder-7B-Instruct Q4_K_M), CUDA backend, greedy (temp 0), 256 tokens, shared 12-prompt suite for tok/s. Context lengths: 4096, 16384, 32768. Flash-attention via -fa on (llama.cpp; flag name has varied across builds — see bench.py comment). VRAM = nvidia-smi used-memory delta on model load (approximate; best on an otherwise-idle GPU). Numbers vary with GPU, driver, and llama.cpp version.
